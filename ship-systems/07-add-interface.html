Module 07: Add Interface

What You Are Building

A user-facing interface for your AI system. By the end of this module, someone other than you will be able to use your system without touching the code. You will build one of three interface types: a CLI, a web UI, or an API endpoint.

Why It Matters

Your system works in a test harness. That is useful to you, but useless to everyone else. An interface is the boundary between your code and the outside world. Without one, your system is a script on your laptop. With one, it is a tool someone can actually use. The interface is what makes the difference between a project and a product.

Step-by-Step Instructions

Step 1: Choose Your Interface Type

Pick one interface type. Do not build two. Use the table below to decide.

Interface Choose This When Recommended Tools
CLI Your user is technical. The system is a tool or automation. Output is text. Python: argparse, click / Node: commander, inquirer
Web UI You want a visual demo. The system has visual output or conversational input. Python: Streamlit, Gradio / Node: Next.js, plain HTML + fetch
API Endpoint The system will be consumed by other software or needs programmatic access. Python: FastAPI / Node: Express, Hono

If you are unsure, pick Web UI. It is the easiest to demo and the most visually compelling for the demo recording in Module 09.

Step 2: Create the Interface File

Create a new file in your project for the interface. Name it based on your choice:

  • CLI: cli.py or cli.js
  • Web UI: app.py (Streamlit/Gradio) or app/page.tsx (Next.js) or index.html
  • API: server.py or server.js

Step 3: Build the Interface

Follow the path that matches your choice. Each path includes starter code you can copy and adapt.

Path A: CLI

Build a command-line interface that accepts input, calls your system, and displays output.

  1. Import your agent/system module
  2. Parse command-line arguments or set up interactive prompts
  3. Call the system with the parsed input
  4. Format and display the output
  5. Add a --help flag with usage instructions
  6. Add error handling that shows useful messages, not stack traces

Python starter (argparse):

#!/usr/bin/env python3
"""CLI interface for [your system name]."""
import argparse
import sys
from agent import run_agent  # import your agent

def main():
    parser = argparse.ArgumentParser(
        description="[Your system] - [one-line description]"
    )
    parser.add_argument("input", help="The input to process")
    parser.add_argument(
        "--format", choices=["text", "json"], default="text",
        help="Output format (default: text)"
    )
    args = parser.parse_args()

    try:
        result = run_agent(args.input)
        if args.format == "json":
            import json
            print(json.dumps(result, indent=2))
        else:
            print(result)
    except Exception as e:
        print(f"Error: {e}", file=sys.stderr)
        sys.exit(1)

if __name__ == "__main__":
    main()

Node starter (commander):

#!/usr/bin/env node
// CLI interface for [your system name]
const { Command } = require("commander");
const { runAgent } = require("./agent"); // import your agent

const program = new Command();

program
  .name("your-system")
  .description("[one-line description]")
  .argument("<input>", "The input to process")
  .option("--json", "Output as JSON")
  .action(async (input, options) => {
    try {
      const result = await runAgent(input);
      if (options.json) {
        console.log(JSON.stringify(result, null, 2));
      } else {
        console.log(result);
      }
    } catch (err) {
      console.error(`Error: ${err.message}`);
      process.exit(1);
    }
  });

program.parse();

Test it by running:

# Python
python cli.py "your test input"
python cli.py --help

# Node
node cli.js "your test input"
node cli.js --help

Path B: Web UI

Build a single-page web interface with an input form, a submit action, a loading state, and an output display.

  1. Set up the page with an input form
  2. Wire the submit button to call your system
  3. Show a loading indicator while processing
  4. Display the output on the page
  5. Handle errors with a visible message

Streamlit starter:

"""Web UI for [your system name]."""
import streamlit as st
from agent import run_agent  # import your agent

st.set_page_config(page_title="[Your System Name]")
st.title("[Your System Name]")
st.write("[One-line description of what the system does.]")

user_input = st.text_area("Input", placeholder="Enter your input here...")

if st.button("Run"):
    if not user_input.strip():
        st.warning("Please enter some input.")
    else:
        with st.spinner("Processing..."):
            try:
                result = run_agent(user_input)
                st.success("Done.")
                st.write(result)
            except Exception as e:
                st.error(f"Error: {e}")

Plain HTML + fetch starter:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>[Your System Name]</title>
  <style>
    body { font-family: sans-serif; max-width: 640px; margin: 40px auto; }
    textarea { width: 100%; height: 120px; }
    #output { white-space: pre-wrap; background: #f5f5f5; padding: 16px; margin-top: 16px; }
    .loading { color: #888; }
    .error { color: #c00; }
  </style>
</head>
<body>
  <h1>[Your System Name]</h1>
  <p>[One-line description.]</p>
  <textarea id="input" placeholder="Enter your input..."></textarea>
  <button onclick="run()">Run</button>
  <div id="output"></div>
  <script>
    async function run() {
      const input = document.getElementById("input").value;
      const output = document.getElementById("output");
      output.className = "loading";
      output.textContent = "Processing...";
      try {
        const res = await fetch("/api/run", {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify({ input })
        });
        const data = await res.json();
        if (!res.ok) throw new Error(data.error || "Request failed");
        output.className = "";
        output.textContent = data.result;
      } catch (err) {
        output.className = "error";
        output.textContent = "Error: " + err.message;
      }
    }
  </script>
</body>
</html>

streamlit run app.py

# Plain HTML (you'll need the API server running too)

Path C: API Endpoint

Build a single API endpoint that accepts JSON input and returns structured JSON output.

  1. Set up a minimal server with one POST endpoint
  2. Parse the JSON request body
  3. Call your system with the parsed input
  4. Return structured JSON output
  5. Return appropriate error codes (400 for bad input, 500 for server errors)
  6. Document the endpoint in your README

FastAPI starter:

"""API endpoint for [your system name]."""
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from agent import run_agent  # import your agent

app = FastAPI(title="[Your System Name]")

class RunRequest(BaseModel):
    input: str

class RunResponse(BaseModel):
    result: str

@app.post("/api/run", response_model=RunResponse)
async def run(request: RunRequest):
    if not request.input.strip():
        raise HTTPException(status_code=400, detail="Input cannot be empty")
    try:
        result = run_agent(request.input)
        return RunResponse(result=result)
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

Express starter:

// API endpoint for [your system name]
const express = require("express");
const { runAgent } = require("./agent"); // import your agent

const app = express();
app.use(express.json());

app.post("/api/run", async (req, res) => {
  const { input } = req.body;
  if (!input || !input.trim()) {
    return res.status(400).json({ error: "Input cannot be empty" });
  }
  try {
    const result = await runAgent(input);
    res.json({ result });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`Server running on http://localhost:${PORT}`);
});

Test it by running:

# Express

#   -d '{"input": "your test input"}'

Step 4: Test with Fresh Eyes

Ask yourself this question: if someone who has never seen your code opened your interface right now, could they figure out what to do within 60 seconds? If the answer is no, simplify. Remove options, add placeholder text, make the default action obvious.

Step 5: Commit Your Interface

git add .
git commit -m "Add user interface ([cli/web/api])"

Done Criteria

CLI

  • Entry point command works (node cli.js or python cli.py)
  • Accepts input via arguments or interactive prompts
  • Displays output in a readable format
  • --help shows usage instructions
  • Errors display useful messages, not stack traces
  • Interface file committed to repo

Web UI

  • Starts with a single command (streamlit run app.py or npm run dev)
  • Input form accepts user data
  • Submit button triggers the system
  • Loading state is visible during processing
  • Output displays clearly on the page
  • Interface file committed to repo

API

  • Server starts with a single command
  • Endpoint accepts POST with JSON body
  • Returns structured JSON response
  • Returns appropriate error codes (400, 500)
  • README documents the endpoint, request format, and response format
  • Interface file committed to repo
