Module 02: Define System

What You Are Building

In this module, you are writing a System Definition—a one-page document that precisely describes what your AI system does, who it serves, what goes in, what comes out, and how you will know it works. Your output is a completed docs/system-definition.md file in your project repository.

Why It Matters

A vague idea produces a vague system. If you cannot describe your AI system in concrete terms—what it takes as input, what it produces as output, and who benefits—you will waste hours building something that does not solve a real problem. The system definition forces clarity before you write a single line of code. Every decision you make in the remaining modules flows from this document. Get it right now and the rest of the course is straightforward. Skip it and you will be reworking your architecture in Module 04.

Step-by-Step Instructions

Step 1: Brainstorm Your System Idea with Claude

Open Claude (or your preferred AI assistant) and use the following prompt to generate system ideas. If you already have an idea, skip to Step 2.

I want to build an AI system that I can complete in about 8 hours.

Constraints:
- It must take a specific input and produce a specific output
- It must solve a real problem for a real user (even if that user is me)
- It must use an AI API (e.g., Claude, GPT) as a core component
- It should be deployable as a CLI tool, web app, or API

Give me 5 specific AI system ideas. For each one, tell me:
1. What it does (one sentence)
2. Who uses it
3. What the input is
4. What the output is
5. Why it is useful

Review the ideas. Pick one that excites you and that you can realistically build. If none of the suggestions fit, modify one or come up with your own—but make sure you can answer all five questions above.

Step 2: Answer the Six Definition Questions

Before filling out the template, answer these questions in plain language. Write one or two sentences for each. Do not overthink it.

  1. What problem does this system solve? Describe the pain point in concrete terms.
  2. Who is the user? Be specific. Not "everyone"—one type of person.
  3. What does the user give the system? What is the exact input? A file? A URL? A text prompt? A form?
  4. What does the system give back? What is the exact output? A report? A file? A response? A decision?
  5. How do you know it worked? What does success look like? Be measurable.
  6. What is out of scope? What will this system NOT do? Name at least two things.

Step 3: Sharpen Your Answers with Claude

Take your draft answers and paste them into Claude with this prompt:

I am defining an AI system I plan to build. Here are my draft answers
to the key definition questions:

[PASTE YOUR ANSWERS HERE]

Help me sharpen these. For each answer:
- Is it specific enough to build from?
- Is the scope realistic for one person to build in 8 hours?
- Are there any ambiguities I should resolve?

Push back on anything vague. Suggest more concrete alternatives.

Update your answers based on the feedback. The goal is precision—anyone reading your system definition should be able to understand exactly what you are building without asking clarifying questions.

Step 4: Fill Out the System Definition Template

Copy the artifact template below into docs/system-definition.md in your project folder. Fill out every field using your sharpened answers.

Step 5: Validate with the Vagueness Test

Read your completed system definition out loud. For each field, ask: "Could two different developers read this and build the same thing?" If the answer is no, the definition is too vague. Rewrite until the answer is yes.

Artifact: System Definition Template

Copy this into docs/system-definition.md and fill out every field:

# System Definition

## System Name
[A clear, descriptive name for your AI system]

## Problem Statement
[What problem does this system solve? Describe the specific pain point
in 2-3 sentences. Be concrete.]

## Target User
[Who is the primary user? Describe one specific type of person,
their role, and why they have this problem.]

## User Scenario
[Walk through one specific scenario: the user has a problem,
they use your system, and the problem is solved.
Write it as: "When [user] needs to [task], they [use system]
and get [result]."]

## Inputs
[List every input the system accepts. Be exact about format.]
- Input 1: [description, format, example]
- Input 2: [description, format, example]

## Outputs
[List every output the system produces. Be exact about format.]
- Output 1: [description, format, example]
- Output 2: [description, format, example]

## Success Criteria
[How do you measure whether the system works? List 3-5 measurable
criteria.]
1. [Criterion 1]
2. [Criterion 2]
3. [Criterion 3]

## Scope: What This System Does NOT Do
[List at least 3 things that are explicitly out of scope.]
1. [Out of scope item 1]
2. [Out of scope item 2]
3. [Out of scope item 3]

Example Output

Here is a completed system definition for an invoice processing system:

# System Definition

## System Name
Invoice Data Extractor

## Problem Statement
Freelancers and small business owners receive invoices in PDF format
from dozens of different vendors. Manually entering invoice data into
spreadsheets takes 5-10 minutes per invoice and is error-prone.
This system automates extraction of key invoice fields.

## Target User
A freelance consultant who receives 20-50 invoices per month from
various clients and vendors, and needs to track them in a spreadsheet
for accounting purposes.

## User Scenario
When a freelancer receives a PDF invoice from a vendor, they upload
it to the system and get back a structured JSON object with all key
fields extracted (vendor name, date, line items, total, due date).
They paste this into their tracking spreadsheet in seconds instead
of minutes.

## Inputs
- Input 1: PDF file, single invoice, any standard format (max 5 pages)
- Input 2: Optional — output format preference (JSON or CSV), string

## Outputs
- Output 1: Structured data with extracted fields, JSON format
- Output 2: Confidence score for each extracted field, 0-100 integer

## Success Criteria
1. Extracts vendor name, date, and total correctly for 90%+ of invoices
2. Processes a single invoice in under 30 seconds
3. Returns structured JSON that can be directly imported into a spreadsheet
4. Handles at least 3 different invoice layouts without custom configuration

## Scope: What This System Does NOT Do
1. Does not handle handwritten invoices
2. Does not store or persist invoice data (stateless processing only)
3. Does not integrate directly with accounting software (QuickBooks, etc.)
4. Does not process multi-currency conversion

Done Criteria

  • docs/system-definition.md exists in your project repository
  • Every field in the template is filled out with specific, concrete content
  • The problem statement describes a real pain point, not a hypothetical one
  • Inputs and outputs are described with exact formats and examples
  • Success criteria are measurable (numbers, not feelings)
  • At least 3 out-of-scope items are listed
  • You can read the definition aloud and another developer would understand exactly what to build
