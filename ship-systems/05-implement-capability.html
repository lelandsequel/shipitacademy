Module 05: Implement Capability

What You Are Building

A working core capability: one complete end-to-end flow that takes real input, processes it through your agent and tools, and produces real output. When you finish this module, your system does the one thing it was designed to do — for real, not with stubs.

Why It Matters

An agent that starts but does nothing is a scaffold. An agent that does one thing end-to-end is a system. This module is where your project crosses that line. You are not trying to implement every feature — you are proving that the core flow works with real data, real API calls, and real output. Once this single flow is solid, everything else you add later is incremental. If you cannot demonstrate one working flow, nothing else matters.

Step-by-Step Instructions

Step 1: Identify Your Core Flow

Before writing any code, answer this question: if your system could only do one thing, what would it be? Write it as a single sentence using this format:

"User provides [input], system [processes it using tools X and Y], and produces [output]."

To find your core flow, ask yourself:

  • What is the flow that proves this system works?
  • What would you show in a 30-second demo?
  • Which flow uses the most critical tools from your architecture spec?

Write your core flow sentence down. You will refer to it throughout this module.

Step 2: Write Your Core Flow Sentence

Open a new file or add to your docs: docs/core-flow.md

# Core Flow

User provides [describe the exact input],
system [step 1 using tool A] then [step 2 using tool B],
and produces [describe the exact output].

## Example Input
[Paste or describe a real example input you will test with]

## Expected Output
[Describe what the output should look like]

Step 3: Implement Tools in Data Flow Order

Work through the flow from left to right — the order data moves through your system. Do not jump ahead. Do not implement the last tool first.

Rules for this step:

  1. One flow only. Do not try to implement everything. Only the core capability.
  2. End-to-end means end-to-end. Input goes in, output comes out. No manual steps in the middle.
  3. Real data, not mock data. Use actual API calls, actual model responses, actual file operations.
  4. Working beats clean. Refactor later. Get it working now.

For each tool in your flow, follow this sequence:

A. Replace the stub with real logic

Open the tool stub file you created in Module 04. Remove the placeholder return and write the actual implementation. Keep it simple — handle the happy path first.

B. Test the tool in isolation

Before wiring it into the agent, confirm the tool works on its own:

Python

# Quick test at the bottom of the tool file (remove later)
if __name__ == "__main__":
    result = search(query="test query here")
    print(result)

Node.js

// Quick test at the bottom of the tool file (remove later)
if (require.main === module) {
  const result = search({ query: "test query here" });
  console.log(result);
}

Run the file directly: python src/tools/search.py or node src/tools/search.js. Does it return real data? If yes, move on. If no, fix it before continuing.

C. Wire it into the agent flow

The tool is already registered from Module 04. Confirm the agent calls it correctly by running the agent and providing input that should trigger this tool.

D. Test the integration

Does data flow correctly from the previous tool into this one? Check that the output format of tool A matches the expected input format of tool B.

Repeat A through D for every tool in your core flow, in order.

Step 4: Run the Full Flow

Start your agent and provide real input:

# Python
python src/agent.py

# Node.js
node src/agent.js

Provide the example input from your core flow document. Watch the output. Check it against the expected output you wrote in Step 2.

Step 5: Debug and Fix

If the output is wrong or the flow breaks, add logging to see what each tool receives and returns:

Python

def search(query: str) -> str:
    print(f"[DEBUG search] Input: {query}")
    result = # ... your real logic ...
    print(f"[DEBUG search] Output: {result}")
    return result

Node.js

function search({ query }) {
  console.log(`[DEBUG search] Input: ${query}`);
  const result = // ... your real logic ...
  console.log(`[DEBUG search] Output: ${result}`);
  return result;
}

Run the flow again. Read the debug output. The bug is almost always in the data handoff between tools — one tool returns a format the next tool does not expect.

Step 6: Verify and Commit

Run the full flow one more time from a clean start. Confirm the output matches your success criteria. Then commit:

git add -A
git commit -m "feat: implement core capability — [describe what the flow does]"

Example: A Research Summarizer Agent

Here is a concrete example of a core flow being implemented, to illustrate the process.

Core flow sentence:

"User provides a research topic, system searches for recent articles, extracts key findings, and produces a structured summary."

Tool implementation order:

  1. search — calls a web search API with the topic, returns a list of article URLs and snippets
  2. extract — takes article URLs, fetches the content, pulls out key findings
  3. summarize — takes the extracted findings, produces a structured summary with sections

Example run:

$ python src/agent.py

You: Summarize recent research on sleep and memory consolidation

Agent: ## Research Summary: Sleep and Memory Consolidation

### Key Findings
1. Slow-wave sleep enhances declarative memory consolidation
   through hippocampal-cortical dialogue (Walker et al., 2024)
2. REM sleep is critical for procedural and emotional memory
   processing (Rasch & Born, 2024)
3. Sleep spindles during NREM stage 2 predict next-day recall
   performance (Mednick et al., 2024)

### Sources
- "The role of sleep in memory consolidation" — Nature Reviews
- "Sleep spindles and memory: new findings" — Journal of Neuroscience
- "REM sleep and emotional memory" — Current Biology

Troubleshooting

Symptom Likely Cause Fix
Tool works alone but fails in agent Input/output format mismatch Log what the agent sends vs. what the tool expects. Check JSON parsing.
API returns errors Auth, rate limits, or malformed request Check API docs, verify your key, inspect the request payload with logging.
Output is wrong but no errors Prompt issue or data transformation bug Log intermediate steps. Inspect what each tool receives and returns.
System is too slow Too many API calls or large payloads Reduce input size for testing. Optimize after the flow works.
Agent calls wrong tool or skips tools System prompt is unclear Make the system prompt more explicit about which tools to call and in what order.
Scope feels too big It is too big Cut features. Implement only the one core flow. Everything else waits.
Tool returns data but agent ignores it Tool response format issue Ensure the tool returns a string. The agent reads the tool's return value as text.

Pacing Guide

Use these checkpoints to gauge whether you are on track:

Time Elapsed Where You Should Be If You Are Behind
30 minutes First tool implemented with real logic, tested in isolation Your scope is too big. Simplify the tool.
60 minutes Data flowing between at least two tools Debug the integration. Check data formats between tools.
90 minutes Core flow produces output (even if rough) Cut scope further. Get any output working.
120 minutes Core flow is complete and repeatable Stop adding features. Fix what is broken and commit.

Done Criteria

  • Core flow identified and written as a single sentence
  • All tools in the core flow implemented with real logic (no stubs)
  • Each tool tested in isolation with real data
  • Tools wired into agent and called in the correct order
  • Full flow runs end-to-end without manual intervention
  • Output matches your success criteria from the system definition
  • Debug logging added (can be removed later)
  • Code committed: git commit -m "feat: implement core capability"
